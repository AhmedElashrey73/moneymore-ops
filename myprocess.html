How to use the five projects
without breaking the system

Which project to open, what to bring into it, what it produces, how sessions should be structured, and how everything connects. Read this before starting any project session.

The five projects — one job each
Project 1
Product OS
Produces all product documentation — Feature Briefs, Fix Briefs, Design Briefs, PRDs, Engineering Tickets, User Stories.
Documentation factory
Project 2
Product Intelligence
Challenges thinking, identifies gaps, analyses evidence, informs backlog decisions. Strategic thinking partner.
Challenge + refine
Project 3
Prototype Lab
Builds interactive HTML prototypes from Design Briefs. Visual thinking layer between documentation and Figma.
Visual thinking tool
Project 4
Analytics & Growth
Owns Mixpanel, CleverTap, and the analytics stack. Interprets data, configures campaigns, maintains event taxonomy.
Analytics brain
Project 5
Copy & Voice
Defines the MoneyMore brand voice and produces all copy across every channel in English and Arabic.
Words layer
Full feature flow — start to sprint
Step 1
Challenge the brief
Project 2
Bring your rough brief or initial thinking into Project 2. Ask it to challenge the problem statement, identify gaps, flag missing edge cases, and question whether the solution traces back to a real user problem. One feature per session. End the session by asking for a structured summary of what was decided and what open questions remain.
Session anchor: "This session is for challenging the [Feature Name] brief only." Paste rough brief. Ask for challenge and structured summary at the end.
↓
Step 2
Write the Feature Brief
Project 1
Take the Project 2 structured summary into Project 1 and produce the formal Feature Brief document. The Feature Brief covers: problem statement, why now, proposed solution, user segments affected, key flows, edge cases and empty states, out of scope, copy moments as [COPY: description] placeholders, Mixpanel events needed, and success metric.
Session anchor: "This session is for the [Feature Name] Feature Brief only." Paste Project 2 summary. One document per session.
↓
Step 3
Write the Design Brief
Project 1
Open a new Project 1 session. Paste the Feature Brief. Produce the Design Brief — every screen, every state, every platform difference, all copy moments as [COPY: description] placeholders. The Design Brief is what the designer and prototype receive. At the end, ask Project 1 to extract all [COPY: description] placeholders and format them as a copy brief list ready for Project 5.
Session anchor: "This session is for the [Feature Name] Design Brief only." New session — do not continue from Feature Brief session.
↓
Step 4
Write the copy
Project 5
Take the copy brief list extracted from the Design Brief into Project 5. Do not upload the full Design Brief document — paste only the copy moments list as plain text. Project 5 produces three variants per copy moment, includes character counts for push notifications, flags which moments need Arabic native speaker review, and notes the segment for each. You approve the final variant for each moment.
Session anchor: "Here are the copy moments for [Feature Name]. Write the copy for each." Paste copy brief list as text. No file upload needed.
↓
Step 5
Update Design Brief with final copy
Project 1
Open a new Project 1 session. Paste the Design Brief and the approved copy from Project 5. Ask Project 1 to replace all [COPY: description] placeholders with the finalised copy. The output is the complete Design Brief with real copy — this is what goes to the designer and to Project 3.
This session is short — paste brief, paste approved copy, replace placeholders. Done.
↓
Step 6
Build the prototype
Project 3
Take the complete Design Brief (with finalised copy) into Project 3. One feature per session. Project 3 builds the HTML prototype using the design system template — Android and iOS frames, all screens and states, real copy, Mixpanel event annotations. You review and approve the prototype. The designer receives both the Design Brief and the prototype as reference.
Session anchor: "Build me a prototype for [Feature Name]. Here is the complete Design Brief." One prototype per session.
↓
Step 7
Designer produces Figma
Designer
Designer receives the complete Design Brief with real copy and the HTML prototype as reference. Figma is produced with real copy from day one — no placeholder text, no second round of copy updates. You review Figma against the Design Brief criteria — not personal taste.
↓
Step 8
Write Engineering Ticket + User Story
Project 1
Open a new Project 1 session. Paste the approved Design Brief and the Figma link. Project 1 writes the Engineering Ticket with full acceptance criteria, finalised copy already in place, Figma link, and definition of done. Then writes the User Story. These go into Jira — Epic linked, label applied, story points estimated at pre-sprint planning.
Session anchor: "Write the Engineering Ticket and User Story for [Feature Name]. Here is the Design Brief and Figma link." New session.
↓
Step 9
Sprint + Release
Engineering
Engineering implements from the ticket. Ticket has everything — accepted criteria, Figma link, finalised copy, definition of done, Mixpanel events to fire. Release readiness checklist is the final gate before anything ships. Sprint release, hotfix, or major feature release — the type determines which checklist applies.

Session rules — how to use projects without overloading them
📌One feature, one session — always
Never bring two features into the same project session. Finish Sign Up completely, then open a new session for Home.
One document type per session. Do not produce a Feature Brief and a Design Brief in the same session.
Start every session with a one-line anchor: "This session is for the [Feature Name] [Document Type] only."
If a session runs more than 30 exchanges, it is too broad. Stop and split.
💡Project 1 — production only, not exploration
By the time something enters Project 1, the thinking is done. Project 1 executes documentation — it does not challenge strategy.
Always paste only what is needed for that session. If writing Engineering Tickets, paste only the Design Brief for that feature.
If the input does not follow the Feature Brief structure, Project 1 will ask for missing sections. Fill them before proceeding.
Mark every user-facing copy moment as [COPY: description]. Never write final copy in Project 1.
🧠Project 2 — thinking only, not writing
Project 2 ends every session with a structured summary — not a document. That summary is the input to Project 1.
Bring rough briefs, screenshots, bullet points — anything. Project 2 works from whatever you bring.
Do not ask Project 2 to write Feature Briefs or Design Briefs. That is Project 1's job.
For Sign Up and Home where screenshots exist — share both the brief and the screenshots. Visual context produces sharper challenges.
✍️Project 5 — paste text, not files
Never upload the full Design Brief to Project 5. Paste only the copy moments list as plain text.
Copy moments are extracted by Project 1 at the end of the Design Brief session — formatted and ready to paste.
Project 5 always produces three variants per copy moment. You approve one. Never one variant, never skip approval.
Arabic is always flagged for native speaker review. Never ship Arabic copy without that review step.
🔬Project 3 — one prototype per session
Only receives the complete Design Brief with finalised copy already in it. Never a brief with [COPY: placeholders] still in it.
One prototype per session. Never two features in the same session.
Always uses the design system template — does not invent new visual conventions.
The prototype is a thinking and communication tool, not a production handoff. Designer still produces the final Figma.
📊Project 4 — specific tasks, not open analysis
Always open with a specific task: "Build the budget creation funnel in Mixpanel" not "analyse Mixpanel."
For Claude in Chrome Mixpanel sessions — use the structured five-task prompt. Do not deviate mid-session.
Short, specific, task-oriented sessions. Project 4 is not a strategy conversation — that is Project 2.
Always check whether the events a campaign depends on are live in Mixpanel before configuring CleverTap.
