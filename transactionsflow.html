Prototype Designer brief Android Draft v1 DB-Transactions April 2026
Transactions Tab — Find & Verify
The list of every money movement, designed for finding a specific transaction and verifying it landed correctly. Search by merchant or amount, filter by type and date, with row-level states for pending categorisation, refunds, and internal transfers. Includes manual add, edit, delete with undo, and category creation.
The problem
Users come to this tab with a job: find a transaction they remember and confirm it's accurate. Today the list is undifferentiated — every row looks the same regardless of source, status, or type, and there is no way to filter or search. Internal transfers inflate the spending picture, refunds look identical to spending, and pending categorisation is invisible. The tab needs to behave like a tool, not a feed.

Accessibility — what changes what you draw
  • Touch targets — 44×44px minimum on every interactive element. Filter chip dismiss × icons, type tab pills, search clear icon, network error banner, six-month cap CTA, mode chip, internal transfer toggle, undo toast action.
  • Contrast — 4.5:1 minimum for text, 7:1 for financial figures. Summary bar values, transaction amounts on rows and detail screen, and EGP currency cues all need contrast tool checks.
  • Never colour-only for state. Internal transfer rows render amounts in neutral grey — colour alone cannot carry the "not counted" meaning. Append the screen reader caption "Internal transfer — not counted as spending or income" after the amount, between amount and time-of-day in reading order.
  • Pending categorisation — screen reader cue. Visible string is "Categorising…" but the screen reader announces "Categorising, please wait" so non-sighted users get the wait cue.
  • Mode chip is a visual cue, not the only one. The Details / Editing chip flips colour and label, but the Save changes / Cancel button pair is the primary signal. Don't rely on the chip alone to communicate edit mode — screen readers should announce mode change on entry to edit mode.
  • Undo toast — 5-second window, screen reader announce on appear. The toast must be announced to screen readers as soon as it appears, including the time remaining (or at minimum that an undo is available). Tapping anywhere outside the toast does not dismiss it; only Undo or timeout closes it.
  • Delete confirmation — destructive action treatment. Delete button uses red background (var(--red)). Cancel comes second in tap order but visually below — destructive action is never the default focus.
  • Focus order — top to bottom on every screen, including RTL. Search → type tabs → filter chips → date pill → summary bar → list. On Arabic the visual direction flips but focus order stays top-to-bottom. In Detail sheet: header → amount → fields top-to-bottom → toggle → action buttons.
  • Row reading order: merchant → category → amount → screen reader caption (internal transfer only) → time-of-day.
