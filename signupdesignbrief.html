Sign-up & sign-in flow — Android
Unified onboarding for new users (Path A) and returning users on a new device (Path C). Same Entry screen for both; routing determined post-auth. Path B (same-device session persisted) routes direct to home with no designed screens. iOS parked.
Problem
One shared flow serves two users with different needs. A new user needs trust built before the SMS ask. A returning user needs continuity — a fast path in and a moment of recognition. The SMS permission remains load-bearing for both: it decides whether the product works automatically or the user falls back to manual. Language, OTP channel, and Hint API must feel like small helpful touches, not extra steps.
Flows in this document
Flow 1 — Path A (new user, includes language switch demo)
Flow 2 — Path C (returning user, Google SSO path)
Flow 3 — Permission denied fallback (A + C)
Flow 4 — Re-prompt surfaces (SMS re-prompt 1, push contextual)
Edges — standalone states (errors, variants, language fallbacks)
Path B — no designed screens (session persisted → home)

Accessibility — what changes what you draw
  • Touch targets — 44×44px minimum on every interactive element. Includes the language trigger, resend link, switch-channel link, "Don't ask again" text link on re-prompts, dismiss actions on bottom sheets, and icon buttons.
  • Contrast — 4.5:1 minimum for text, 7:1 for financial figures (amounts, balances). Check every frame with a contrast tool.
  • Never colour-only for state. Disabled CTAs (OTP channel CTA before selection, SMS pre-prompt skip button), error inputs, countdown states on the resend link, and selected-channel state on the OTP sheet all need an icon or text signal alongside colour.
  • Focus order — input → primary CTA → secondary actions. Logical top-to-bottom. On Arabic layouts the visual direction flips to RTL but focus order stays top-to-bottom, not visual-left-to-right.
  • Push contextual prompt is never timed-dismissible. Screen reader users need time to read and decide — no auto-dismiss, no countdown.
Must-not reminders
Load-bearing design guardrails for this flow. Check every frame against them before review.
  • Do NOT make the SMS pre-prompt or the push contextual prompt feel like a sales pitch. No emotive language, no exclamation marks, no stacked value props. They are trust screens.
  • Do NOT demote the skip on the SMS pre-prompt or the push contextual to a barely-visible text link. Denying must feel like a legitimate path — visible button, not a subtle link.
  • Do NOT visually demote either option on the Entry screen. Google SSO and phone are equal-weight. No "recommended" tag. No default selection. No size hierarchy.
  • Do NOT pre-select a default OTP channel. The CTA stays disabled until the user actively chooses SMS or WhatsApp.
  • Do NOT silently fall back between OTP channels or from Google SSO to phone. The user sees the error and makes an active choice — retry same channel, or switch.
  • Do NOT auto-send OTP after a Phone Number Hint selection. The Hint fills the field; the user must tap Send code explicitly.
  • Do NOT design a fixed-duration loading screen. Three outcomes: success (parseable count greater than zero), "primed and ready" (count = zero), timeout empty state. Three distinct screens.
  • Do NOT show Welcome back to new users (Path A). It is Path C only. Do NOT show Name capture to Google SSO users or Path C users — it is Path A phone only.
  • Do NOT make the SMS or push re-prompts full-screen takeovers. Bottom sheets, dismissible, clearly distinguishable from OS-level prompts.
  • Do NOT surface the language menu on post-auth screens. Onboarding-only affordance.
  • Do NOT flash branding behind the Google consent sheet or the Phone Number Hint sheet. In-flight states are neutral.
