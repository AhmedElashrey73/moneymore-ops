Proposal 1 — personalised offers

A two-layer personalisation model

Rather than treating personalisation as a single activity with a binary risk profile, we split it into two distinct layers. Layer 1 operates on in-app behaviour only — no SMS data, no secondary purpose concern. Layer 2 uses SMS transaction data with explicit opt-in. Both layers can coexist. Neither forces the other.

Layer 1 · Active from day one
In-app behaviour
Data source: Offers section activity only
Personalisation based entirely on what the user does within the Offers section — what they tap, browse, and redeem. No SMS transaction data involved at any point. The user's reasonable expectation is already aligned with this use.
Privacy Policy disclosure only
No separate consent screen required. Directly avoids the secondary purpose concern raised in R-3.
Layer 2 · Explicit opt-in required
SMS transaction data
Data source: Bank SMS transactions (sensitive data under PDPL)
For users who want a deeper experience — offers matching their real spending patterns — we offer a voluntary upgrade. This consent is separate, granular, and independent from general app consent and the SMS access grant. Users who decline still get Layer 1.
Explicit granular opt-in required
Directly addresses R-1 (legal basis), R-2 (valid consent), R-3 (purpose limitation) with an auditable consent record.
Core principle: No data crosses between layers without explicit consent. SMS transaction data used in Layer 2 is never combined with Offers behaviour data from Layer 1 unless the user has independently consented to both.

Why this screen works legally
The user sees a dedicated screen that cannot be skipped or bundled. The consent is informed (clear explanation), specific (named purpose: offer personalisation), and freely given (genuine binary choice with no penalty for declining). The affirmative CTA creates an auditable consent record.
Why we surface it here
The opt-in appears immediately after SMS access is granted — when the user most understands what their transaction data is. Surfacing it later (e.g. first Offers tab visit) breaks the moment and feels out of context.
Question for DPO — Q1
Is in-app offers behaviour — what a user taps and redeems within the Offers section — classified as sensitive data under PDPL in the same way SMS transaction data is? This determines whether Layer 1 requires a consent gate or whether Privacy Policy disclosure is sufficient.
Proposal 2 — rotating savings group integration

Elgameya — lead introduction model

Rather than positioning MoneyMore as a data-sharing partner to Elgameya, we act purely as a lead introduction engine. MoneyMore surfaces the offer, collects four intent signals from the user through a one-question-per-screen flow, and redirects with a promo code. No SMS data, no spending history, no financial profile is ever shared.

1
User sees Elgameya in the Offers tab
Elgameya is featured as a card in the Offers section. The card communicates what Elgameya offers and that a special promo code is available for MoneyMore users.
2
User taps to explore
A dedicated Elgameya screen appears. Description of the service, eligibility, and the available promo code. No data collected at this point.
3
Four intent questions — one per screen
The user answers four questions across four dedicated screens with a progress bar throughout. These serve two purposes: routing the user to the right Elgameya gameya, and giving MoneyMore signal about user financial goals.
Screen 1: What is your goal for joining a gameya?
Screen 2: How much money are you looking to collect?
Screen 3: When are you planning to enter?
Screen 4: Which seat would you like to reserve?
4
Natural consent moment — before redirect
A dedicated consent screen lists exactly what will be shared with Elgameya before the user taps to proceed. The CTA is "Get my promo code" — reward-framed, not legal-framed.
The screen clearly states what travels to Elgameya (the four answers and a promo code) and what does not (transaction history, spending data). MoneyMore's retention of the answers for product understanding is also disclosed here.
5
Redirect to Elgameya with promo code
User is redirected to Elgameya. The four answers and the promo code travel with the redirect. Elgameya conducts their own onboarding under their own legal basis and privacy policy. MoneyMore's involvement ends at the redirect.

Question for DPO — Q2
Does the natural consent approach built into the consent screen satisfy PDPL requirements for sharing the four answers with Elgameya? Or does a more formal separate consent screen need to appear before the questions begin?
Competitive landscape — please review
Primary: Telda, Klivver — closest to MoneyMore's model
Secondary: Waffarha, Lucky — relevant for offers handling
Are there PDPC-accepted consent patterns any of these have used that we can reference?
