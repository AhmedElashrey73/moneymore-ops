The three release types

Not every release is the same

MoneyMore has three distinct types of release. Each has a different trigger, a different checklist, and a different level of urgency. Understanding which type you are doing determines everything else.

🚂
Sprint release
The standard release at the end of every sprint. The train leaves on schedule regardless of what is and isn't ready. Unfinished tickets get pulled — they don't delay the train. This is the most common release type.
When: Every sprint end · Every ~2 weeks · Scheduled · No exceptions
🔥
Hotfix release
An unplanned release that happens outside the sprint cycle. Only valid for critical bugs that are breaking the app for users right now and cannot wait until the next sprint. Not for nice-to-haves. Not for pressure. Only for critical.
When: Critical bug live in production · Must be approved by Head of Product before starting
✨
Major feature release
A sprint release that contains a significant new user-facing feature. Uses the full sprint checklist plus additional gates for user experience, campaign readiness, and staged rollout. Happens as part of the normal sprint cycle — not a separate event.
When: Sprint contains a new feature that changes the user experience meaningfully

Next sprint planning
The train rule: On Day 10, the release goes out with whatever is Done. Tickets that are not Done get pulled from the release and moved to the next sprint backlog. The train does not wait. This is intentional — it creates a forcing function for the team to finish things properly rather than rushing incomplete work into a release.

Hotfixes happen between trains only when something is critically broken for users right now — app unusable level. Head of Product must approve before engineering starts the hotfix. Everything else waits for the next train.

Staged rollouts — no custom engineering required

How to release to a percentage of users first

Both Google Play and the App Store have built-in percentage rollout tools. You do not need to build anything custom. If a release breaks something, it breaks for 10% of users instead of 100%. This is especially important for MoneyMore since a bad release could corrupt SMS transaction data or break OTP for everyone simultaneously.

Google Play · Android
How to set a staged rollout on Google Play
1
Go to Google Play Console → select MoneyMore app
2
Production → Releases → Create new release
3
Upload the AAB file and fill in the release notes
4
Under "Release to production" — choose "Set rollout percentage"
5
Set to 10%. Submit for review. Once approved, monitor Crashlytics for 48 hours.
6
Increase percentage manually: 25% → 50% → 100% every 2 days if no issues
⚠ You can halt a staged rollout at any percentage if a crash spike appears. Go to the release in Play Console and tap "Halt rollout." Users who already updated keep the new version.
App Store · iOS — when iOS launches
How to set a phased release on App Store Connect
1
Go to App Store Connect → MoneyMore → prepare new version
2
Fill in version information and release notes
3
Under "Version Release" — select "Automatically release this version" but enable "Phased Release"
4
Apple handles the percentage rollout automatically: 1% → 2% → 5% → 10% → 20% → 50% → 100% over 7 days
5
Monitor Crashlytics daily during the rollout period
6
If a critical issue appears, pause the rollout in App Store Connect → Version → Pause Phased Release
ℹ Apple's phased release applies only to users with automatic updates enabled. Users who manually update always get the latest version regardless of rollout percentage.
When to skip staged rollout and go straight to 100%
For hotfix releases fixing a critical bug — go straight to 100% because you need to fix it for everyone immediately. For sprint releases with only bug fixes and no new user-facing features — 100% is also acceptable. Staged rollout is most important when you are shipping a new feature or making changes to core flows like sign-up, SMS reading, or transaction categorisation.
Release note templates

What to write and who writes it

Every release needs two documents. Product writes the user-facing release note — what users see in the App Store and Google Play. Engineering writes the internal technical changelog — what the team uses to understand what changed and why. Neither is optional.

User-facing release note — written by Head of Product · Published on App Store and Google Play
Version [X.X] — [Month Year] — If this is a bug fix sprint, lead with what was broken and is now fixed — We fixed [plain language description of the main bug — e.g. "an issue where your budget summary wasn't loading on the home screen"]. We also fixed [second bug if applicable — keep it to one sentence].
— If this is a feature sprint, lead with what is new — You can now [plain language description of the new feature — e.g. "set monthly spending limits for each category and we'll track your progress automatically"]. We also improved [supporting change if applicable].
— Always end with this line — Thank you for using MoneyMore. If you run into anything, reach out to us at [support email].
Rules: No technical language. No mention of APIs, endpoints, or engineering terms. Write for a user who has never read a release note before. Maximum 5 sentences total. Arabic translation required before publishing.
Internal technical changelog — written by Engineering Lead · Shared in Slack with the team
Release [X.X] — [Date] — [Sprint name] What shipped Fix Track · [Ticket ID] [Fix name] — [one line on what was wrong and what the fix was] · [Ticket ID] [Fix name] — [one line]
Build Track · [Ticket ID] [Feature name] — [one line on what was built] · [Ticket ID] [Feature name] — [one line]
What did not ship · [Ticket ID] [Name] — [one line on why it was pulled — not ready / blocked / deprioritised]
Known issues going into next sprint · [Any known issue that is live in production that the team should be aware of]
Rollout · Platform: Android [/ iOS when live] · Rollout type: [Staged — starting at 10% / Full — 100%] · Monitoring: Crashlytics and New Relic active · First check at [time] tomorrow · Rollback plan: [What we do if a critical issue appears in the first 24 hours]
Rollback plan — what to do if something breaks after release
If the issue is caught in staged rollout (under 100%): Halt the rollout immediately in Play Console or App Store Connect. Do not increase the percentage. Fix the issue and create a new build.
If the issue is caught at 100% rollout: Assess severity first. App unusable level → issue a hotfix immediately. Broken flow level → assess whether it can wait for the next sprint or needs a hotfix. Degraded UX level → add to Fix Track for next sprint.
Google Play rollback: You can roll back to the previous version in Play Console → Production → previous release → "Re-release." This is faster than submitting a new hotfix build.
App Store rollback: Apple does not allow rolling back to a previous version. The only option is submitting a new hotfix build through expedited review.
