Offers Personalization v2
A consent gate on the Offers tab where users opt into personalization from spending patterns and partner push notifications, with preferences managed from Profile. Two independent preferences, single consent surface used by all entry points, full state management for grandfathered v1 users.
The problem
v1 shows the same offer feed to everyone. v2 introduces personalization based on spending patterns (read from SMS) and partner push opt-in — but both require explicit user consent. The design challenge is making consent feel like a clear, fair trade rather than a wall, while supporting four distinct entry contexts: new user first visit, existing v1 user, Profile-driven activation, and Profile-driven disable.

Accessibility — what changes what you draw
  • Locked state — gift icon must have a text label for screen readers. The headline serves this purpose; ensure the icon is marked decorative in the implementation.
  • Checkbox state needs both visual and announced cues. Green border + checkmark + tinted background = visual. Screen reader must announce "checked" / "unchecked" on tap.
  • Toggle state announcements. "Personalize my offers, on" / "Personalize my offers, off" — full label spoken with state.
  • Disabled toggle must be announced as such. "Partner offer notifications, off, dimmed, double-tap to learn why" or equivalent. The inline warning below carries the explanation.
  • Touch targets 44×44px on all CTAs, checkboxes, toggles, the Skip action in the banner, and the system settings link.
  • Privacy Policy link uses underline + arrow icon — not colour alone. Important since legal compliance often requires the link to be visually distinct from body text.
  • Confirm CTA disabled state. Must use both colour change (grey) and announced state (button, dimmed, disabled) — not colour alone.
