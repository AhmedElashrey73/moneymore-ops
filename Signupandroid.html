Sign-up & sign-in flow — Android
Unified onboarding for new users (Path A) and returning users on a new device (Path C). Same Entry screen for both; routing determined post-auth. Path B (same-device session persisted) routes direct to home with no designed screens. iOS parked per FP-030 §12.
Problem
One shared flow serves two users with different needs. A new user needs trust built before the SMS ask. A returning user needs continuity — a fast path in and a moment of recognition. The SMS permission remains load-bearing for both: it decides whether the product works automatically or the user falls back to manual. Language, OTP channel, and Hint API must feel like small helpful touches, not extra steps.
Flows in this document
Flow 1 — Path A (new user, includes language switch demo)
Flow 2 — Path C (returning user, Google SSO path)
Flow 3 — Permission denied fallback (A + C)
Flow 4 — Re-prompt surfaces (SMS re-prompt 1, push contextual)
Edges — 15 standalone states from §4
Path B — no designed screens (session persisted → home)

IF / THEN — all routing logic
Entry + authentication
IF first app launch AND device language = Arabic THEN locale defaults to AR, layout renders RTL.
IF first app launch AND device language ≠ Arabic THEN locale defaults to EN.
IF user taps language trigger THEN show dropdown (EN, العربية), currently-selected indicated by checkmark. On select, flip INSTANTLY — no reload, no confirm modal, no loading state.
IF user on post-auth screen THEN language menu is NOT visible. Language selection persists from onboarding.
IF user taps phone field on Entry for FIRST time THEN fire Hint API sheet.
IF user taps phone field on Entry for second+ time in same screen session THEN do NOT fire Hint API. Focus the field and show keyboard only.
IF Hint API unavailable OR returns zero numbers OR user dismisses THEN manual input field fully functional. NO banner, NO error, NO degraded-experience messaging.
IF user selects number from Hint sheet THEN populate the field. Do NOT auto-send OTP. User must explicitly tap Send code.
IF user taps Continue with Google AND SSO succeeds AND account exists THEN route to Path C (Welcome back → SMS pre-prompt).
IF user taps Continue with Google AND SSO succeeds AND account does not exist THEN route to Path A post-auth (skip Welcome back and Name capture — OAuth has the name → SMS pre-prompt).
IF Google SSO fails (network, cancelled, account issue) THEN return to Entry screen with inline error. NEVER silently fall back to phone.
IF user's Google account already registered on another device THEN show Duplicate account — Google match screen, auto sign-in (Path C behaviour).
OTP
IF user enters phone + taps Send code THEN show OTP channel bottom sheet with no default selection.
IF phone number entered matches existing account THEN show Duplicate account — phone match screen, route to sign-in with phone pre-filled.
IF user has not chosen a channel THEN OTP sheet CTA disabled with lock icon (non-colour signal).
IF selected channel fails (send error, delivery fail, attempts exhausted, rate limit) THEN show inline error with Retry + Switch channel CTAs. NEVER silently fall back between channels.
IF user taps switch-channel link at any point THEN re-send OTP on other channel, restart countdown, update title to reference new channel. Preserve phone number.
IF network request never reached server THEN show network error. Do NOT consume an OTP attempt.
IF rate limit hit (OQ-05 in FP-030) THEN show rate-limit error with "use different number" + "contact support" CTAs. Disable OTP input.
IF user taps back from OTP entry THEN return to Entry with phone number preserved.
SMS permission + scan
IF user on SMS pre-prompt taps Allow SMS access THEN fire OS permission dialog.
IF OS "Don't ask again" state already active THEN do NOT call permission API. Route to Settings deep-link screen.
IF OS permission granted THEN start scan. Show Loading with progressive-reveal messages.
IF OS permission denied (not "Don't ask again") THEN route to Manual-entry fallback → Home in manual_only state.
IF scan completes AND parseable_count > 0 THEN show Celebration.
IF scan completes AND parseable_count = 0 THEN show "primed and ready" Empty state.
IF 15s backend timeout fires THEN show Scan timeout empty state.
IF scan has been running for 6s AND no completion signal THEN hold loading screen at message 3 until scan completes OR backend timeout fires.
Re-prompts + push
IF user in manual_only AND manual tx count = 3 AND re-prompt 1 not yet fired AND no other SMS re-prompt fired this session THEN show SMS re-prompt 1 bottom sheet.
IF user on re-prompt bottom sheet taps Enable SMS THEN fire OS permission dialog DIRECTLY. Do NOT route back to SMS pre-prompt.
IF user on re-prompt bottom sheet taps Enable SMS AND "Don't ask again" state active THEN route to Settings deep-link, not OS dialog.
IF user has dismissed 3 SMS re-prompts cumulatively THEN permanently suppress SMS re-prompts for this user.
IF first transaction display event fires (post-onboarding, SMS-enabled) AND push permission not yet requested THEN show Push contextual prompt overlaying tx list.
IF user on push contextual prompt taps Turn on notifications THEN fire OS push dialog directly.
IF 90-day session expiry OR hard-update OR security event fires THEN on next app open, route to Entry screen as Path C. Clear persisted session.
IF mid-onboarding network disconnect causes partial data sync THEN show Mid-onboarding disconnect restart screen. Log step where disconnect occurred.

§8 must-not-do reminders
These are the most frequent failure modes for flows like this. Check every frame against them.
  • Do NOT make SMS pre-prompt or push contextual feel like a sales pitch. Emotive language and exclamation marks degrade trust.
  • Do NOT demote the skip on SMS pre-prompt or push contextual to a barely-visible text link. Denying must feel like a legitimate path.
  • Do NOT trigger OS SMS dialog on any screen other than SMS pre-prompt or re-prompt's Enable action. Never silently, never on load.
  • Do NOT request push during onboarding. Push is contextual, first-transaction-display only.
  • Do NOT visually demote either option on Entry. Google and phone are equal-weight. No "recommended" tag.
  • Do NOT pre-select a default OTP channel. CTA stays disabled until user chooses.
  • Do NOT silently fall back between OTP channels or from Google SSO to phone. User sees the error, makes an active choice.
  • Do NOT auto-send OTP after Hint API selection. User taps Send code explicitly.
  • Do NOT re-trigger Hint API sheet on subsequent taps. First tap only per screen session.
  • Do NOT surface a blocking error if Hint API is unavailable. Manual field is first-class.
  • Do NOT design a fixed-duration loading screen. Three outcomes based on actual scan result.
  • Do NOT show celebration when parseable_count = 0. Do NOT show "primed and ready" when backend timed out. Three distinct states.
  • Do NOT show Welcome back to new users. Path C only.
  • Do NOT show Name capture to Google SSO users or Path C users. Path A phone only.
  • Do NOT design manual fallback as a failure state. Manual_only users are legitimate users.
  • Do NOT make SMS or push re-prompts full-screen takeovers. Bottom sheets, dismissible.
  • Do NOT surface language menu on post-auth screens. Onboarding-only affordance in v1.
  • Do NOT write or modify copy. All strings defined in §5.
  • Do NOT convey state through colour alone. Icons + text + colour, not colour alone.
  • Do NOT flash branding behind Google consent sheet or Hint API sheet. In-flight states are neutral.
