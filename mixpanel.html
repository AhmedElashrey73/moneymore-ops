Getting started · Chapter 1

What is Mixpanel?

Mixpanel is the tool that tells us what users are actually doing in the MoneyMore app — not what we think they are doing, and not what they say they are doing. It records real user behaviour in real time.

The honest answer in one sentence

Mixpanel records every meaningful action a user takes in the app, and then lets us count, filter, and analyse those actions to understand how people actually use MoneyMore.

Why this matters for your role
Every campaign you configure in CleverTap needs to be triggered by something. That something is a Mixpanel event. If you do not understand what events exist and what they mean, your campaigns will either not fire, fire at the wrong moment, or reach the wrong users. Mixpanel is the foundation that CleverTap is built on.

What Mixpanel is not

Mixpanel is not a customer database — it does not store names, phone numbers, or personal details. It stores anonymous identifiers tied to behaviour. When you look at a user in Mixpanel, you see what they did, not who they are.

Mixpanel is not a real-time dashboard for business KPIs — it is a behavioural analytics tool. For high-level business metrics, that is what our other dashboards are for. Mixpanel is for understanding the how and why behind the numbers.

Getting started · Chapter 2

How the data is structured

Before you can read a Mixpanel report, you need to understand the three building blocks that everything is made from: events, properties, and user profiles. Everything in Mixpanel is a combination of these three things.

Event
A specific action that a user took at a specific moment in time. Every event has a name and a timestamp.
e.g. "Screen Viewed" — fired at 14:23:07 on 10 April 2026
Property
Additional information attached to an event that describes what happened in more detail. An event without properties just tells you something happened. Properties tell you what, where, how, and who.
e.g. Screen Viewed + Screen Name: "BudgetsOverview" + Screen Class: "BudgetsOverviewScreen"
User Profile
A collection of persistent facts about a user that do not change with each event — things like when they signed up, what platform they are on, and how many total sessions they have had.
e.g. platform: "android" · created_at: 1775865037 · $ae_total_app_sessions: 12
The mental model that makes Mixpanel click
Think of it like this. An event is a sentence: "Something happened." A property is a detail that completes the sentence: "A user viewed the Budgets screen." A user profile is the biography of the person who did it.

How these connect in practice

When a MoneyMore user taps the Budgets tab, this is what Mixpanel receives:

The user tapped the Budgets tab in the bottom navigation bar.

Core concepts · Chapter 4

Properties

Properties are the detail layer that makes events useful. Without properties, you know something happened. With properties, you know what happened, to whom, where, and in what context.

Two types of properties

Event properties — attached to a specific event
These describe the specific action. They only exist in the context of the event they belong to. Every event can have different properties.
Budget Created → Category: "Food & Dining" · Budget Amount: 2000
User properties — attached to the user permanently
These describe the user themselves and persist across all their events. When a user property changes, it updates the user's profile everywhere.
platform: "android" · $ae_total_app_sessions: 24 · created_at: 1775865037

Core concepts · Chapter 5

User profiles

A user profile in Mixpanel is the persistent record of everything we know about a specific user across all their sessions. While events are individual moments, the user profile is the ongoing story.

What MoneyMore user profiles contain

Every MoneyMore user profile currently has these properties set automatically:

PropertyTypeWhat it means
platformstringWhich platform the user is on — "android" for all current users
created_atnumberWhen the account was created (Unix timestamp)
is_new_userstringWhether the user is new — should be a boolean, currently stored as a string
$ae_total_app_sessionsnumberTotal number of sessions this user has had — auto-computed by Mixpanel
$ae_total_app_session_lengthnumberTotal seconds this user has spent in the app — auto-computed
$android_app_versionstringThe app version the user is currently on
$android_os_versionstringThe Android version on the user's device
The most important gap in our user profiles
There is no segment property on any user profile. This means we cannot filter any Mixpanel report by salary worker, freelancer, or student. Every report you build today shows all users together with no way to separate them by who they are. This is being fixed in the restructure — but for now, be aware that your reports are always showing the full mixed user base.

How to look at a specific user profile

1
Go to Users in the left sidebar
Click the Users icon (looks like a person). This opens the user explorer.
2
Search for a user by their distinct ID or filter by property
If you have a user's ID from a support ticket, you can search directly. Or use filters to find users who match certain criteria — for example users who have fired Budget Created.
3
View their activity stream
Click on any user to see a chronological timeline of every event they have ever fired in the app. This is the most powerful debugging tool available — you can see exactly what a specific user did, in what order, and what happened.
When to look at individual user profiles
When a customer contacts support with a problem, pull up their user profile in Mixpanel before responding. You will immediately see what they actually did in the app — which screens they visited, whether they completed the action they claim to have completed, and what errors they encountered. This saves significant back-and-forth.
Reading reports · Chapter 6

The Events report

The Events report is the simplest report in Mixpanel. It shows you how many times specific events fired over a time period. This is where you start when you want to answer "how often does X happen?"

How to read an Events report

The Events report shows a line chart or bar chart with time on the X axis and event count on the Y axis. Each line is one event type. The number you see is the total count of times that event fired in the selected time period.

Event count vs unique users — know the difference
An event count tells you how many times the event fired. A unique user count tells you how many distinct users fired it at least once. These are very different numbers. Budget Created firing 170 times could mean 170 different users each created one budget, or it could mean 85 users each created two budgets. Always check which you are measuring.

The five most useful things to track in the Events report

What to trackEvent to useWhy it matters
Are users signing up? Sign Up Completed Your top-of-funnel health check. If this drops, something is broken in onboarding.
Are users importing their SMS? Sms Import Completed The activation milestone. Users who do not import SMS never see their first value.
Are users creating budgets? Budget Created The key retention behaviour. 170 events in 90 days — we need this to grow.
Are there API errors? Api Error 33,686 in 90 days. Filter by Endpoint to see which API is broken. /budgets/monthly is the current critical bug.
Are users opening the app regularly? App Opened 9,330 in 90 days. The baseline engagement signal. Watch for drops.

The Api Error event — your most important debugging tool

The Api Error event deserves special attention. It fires every time the app makes an API call that fails. It carries three critical properties:

Api Error
33,686 fires in 90 days — every backend failure is here
Fires on every failed API call. The three properties below tell you everything about what failed and why.
Endpoint→ "/budgets/monthly" (the URL that failed)
Error Message→ "Resource not found." (what the server said)
Http Status→ 404 (the error code)

To find which API endpoints are failing: open the Events report → select Api Error → break down by Endpoint property. You will immediately see a ranked list of every broken endpoint and how often it is failing.

Reading reports · Chapter 7

Funnels

A funnel shows you how many users complete a sequence of events in order. It answers the question: "Of all the users who did step 1, how many went on to complete step 2, then step 3?"

Why funnels matter for MoneyMore

Funnels reveal where users drop off. A user who signs up but never imports their SMS is lost before they see the app's core value. A user who starts creating a goal but never completes it is a friction problem. Funnels make these invisible drop-offs visible.

How to read a funnel in Mixpanel

1
Look at the conversion rate between each step
Mixpanel shows you the percentage of users who completed each step out of those who completed the previous one. A sudden large drop means something is breaking or confusing users at that step.
2
Check the time to convert
Mixpanel also shows how long users take to move between steps. If users take 3 days to go from sign up to budget creation, that tells you the feature is not being discovered immediately — navigation or onboarding might need improvement.
3
Break down by a property to find patterns
Adding a breakdown by city or device model can reveal if the drop-off is specific to certain groups. If 90% of Cairo users convert but only 40% of Alexandria users do, that is a meaningful geographic signal.
Reading reports · Chapter 8

Retention

Retention tells you whether users come back. A user who signs up once and never returns is not a retained user. Retention is the single most important metric for MoneyMore's long-term health.

How Mixpanel measures retention

Mixpanel's retention report works like this: it takes a group of users who did something on Day 0 (for example, signed up), then tracks what percentage of them came back and did something else (for example, opened the app again) on Day 1, Day 7, Day 14, and Day 30.

The two retention numbers that matter most for MoneyMore
D14 retention — of users who signed up, what percentage opened the app again 14 days later? This is the medium-term retention signal. Our target is 42%.

D30 retention — same but 30 days later. Our target is 30%. If D30 is strong, users have formed a habit. If it is weak, the app is not sticky enough to survive a month.

How to build a retention report in Mixpanel

1
Go to Retention in the left sidebar
The Retention report is usually under the Insights or Analysis section.
2
Set the "Born on" event to Sign Up Completed
This tells Mixpanel: start counting from the day a user signed up.
3
Set the "Retained by" event to App Opened
This tells Mixpanel: count a user as retained if they opened the app again on that day.
4
Read the retention table
Each row is a cohort of users who signed up in a given week. Each column is how far after sign-up you are checking. The percentage in each cell is how many of that cohort were still active on that day.
The most important retention insight in the MoneyMore data
Users who create a budget retain at significantly higher rates than users who do not. This is the single strongest product insight we have. It means every campaign you run to drive Budget Created events is directly contributing to retention — not just engagement.
Building reports · Chapter 9

Your first report

This chapter walks you through building a real report from scratch. By the end you will have built the budget creation report — a report you will actually use every sprint.

The report: how many budgets are being created each week?

This is one of the most important weekly metrics to track. Here is how to build it:

1
Open Insights from the left sidebar
Insights is the main report builder in Mixpanel. Click the chart icon.
2
Click "Add event" and search for Budget Created
Type "Budget" in the search box. Select "Budget Created" from the dropdown.
3
Change the metric from "Total" to "Unique"
Click the dropdown next to Budget Created and change from Total (how many times it fired) to Unique (how many distinct users fired it). This tells you how many users created budgets, not how many budgets were created total.
4
Set the date range to Last 90 days, grouped by Week
The date picker is in the top right. Select Last 90 days. Change the interval dropdown from Day to Week for a cleaner view.
5
Add a filter: Status = success
Click "Filter" and add the property Status equals "success". This removes failed budget creation attempts and shows only successful creations.
6
Save the report
Click Save in the top right. Name it "Budget Created — weekly unique users". Add it to the MoneyMore dashboard.
You have now built a report that matters
This report tells you whether the number of users creating budgets is growing, flat, or declining week over week. If it drops after a release, something in the restructure may have broken the budget creation flow. If it grows after a CleverTap campaign, your campaign is working.
Building reports · Chapter 10

Filtering by segment

Filtering lets you narrow a report to a specific group of users or events. This is where Mixpanel becomes truly useful — not just knowing what happened, but knowing what happened to which users.

Two ways to filter in Mixpanel

Filter by event property
Narrows the events included in the report based on a property attached to that event. Applied within the report builder.
Budget Created where Status = "success" — only shows successful budget creations
Filter by user property (cohort)
Narrows the users included in the report based on a persistent user property. Applied at the user level.
Only show events from users where $android_os_version = "10" — to check if a bug is Android 10 specific
The segment filter we cannot use yet
We cannot filter by salary worker, freelancer, or student because the segment property does not exist on user profiles yet. Until it is added in the restructure, all reports show the full mixed user base. When you present findings from Mixpanel, always note that you cannot currently separate by segment.

Day-to-day work · Chapter 11

The signal review session

Every sprint on Day 5, you sit with Product to review all user signals together. This is the most important regular meeting for your role. Mixpanel is your primary input for this session.

What you are doing in the signal review

You are not just reading numbers. You are identifying patterns. A single data point is trivia. A pattern across multiple signals is a product insight that should affect the backlog.

Your preparation checklist before every signal review

Open Mixpanel and check the Api Error event for the last 7 days. Filter by Endpoint and note any endpoints with unusually high error volume.
Check Budget Created for the last 7 days (unique users). Is it up, flat, or down vs the previous 7 days?
Check Sign Up Completed for the last 7 days. Is new user acquisition healthy?
Check Otp Delivery Failed. Is the rate improving after FIX-003 is deployed?
Check App Opened for the last 7 days vs the previous 7 days. Overall engagement trend.
Note any event that has changed significantly (up or down more than 20%) since the last sprint. These are the signals worth discussing.

How to present a Mixpanel finding

When you bring a Mixpanel finding to the signal review, structure it as:

1
What I observed
State the data fact. "Budget Created fired 28 times in unique users this week, down from 42 last week — a 33% drop."
2
What might explain it
State possible explanations — not conclusions. "This could be related to the /budgets/monthly 404 error making budgets less visible on home. Or it could be a seasonal pattern."
3
What I think we should do
State a recommended action. "I recommend we confirm whether this is related to FIX-001 before treating it as a separate signal."
The rule about drawing conclusions from Mixpanel
Mixpanel tells you what happened. It rarely tells you why. When you present a finding, be clear about which part is data and which part is your interpretation. "The data shows X. I believe this is because Y" is honest. "The data proves Y" is almost never true.
Day-to-day work · Chapter 12

Things to never do

Just as important as knowing how to use Mixpanel is knowing what not to do. These are the most common mistakes that lead to bad decisions.

Never treat a single data point as a trend
If Budget Created spikes to 50 on one day, that is interesting but not a trend. A trend requires at least 3 comparable data points over time. One anomalous day could be a bug, a test account, or a coincidence.
Never confuse event count with unique user count
If App Opened fired 9,330 times, that does not mean 9,330 users opened the app. One user opening the app 10 times counts as 10 events but 1 unique user. Always check whether you are looking at total events or unique users — and always state which one you are reporting.
Never build a CleverTap campaign based on an event you have not verified
Before you configure a CleverTap campaign triggered by a Mixpanel event, confirm the event is actually firing correctly in Mixpanel. Check the event stream. Look at the properties. Confirm the volume makes sense. If you configure a campaign based on a broken event, the campaign will never fire — or will fire for the wrong users.
Never report a finding without noting the date range
"Budget Created is at 170" means nothing without knowing the time period. Always state the date range with every number you report. "Budget Created: 170 unique users in the last 90 days."
Never assume the data is complete
Events can be missing, broken, or firing incorrectly. The taxonomy evaluation shows six events that are live but have broken or missing properties. If a number seems surprisingly low or high, investigate whether the underlying event is instrumented correctly before drawing conclusions.
