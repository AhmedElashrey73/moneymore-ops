Getting started · Chapter 1

What is CleverTap?

CleverTap is where we act on what Mixpanel tells us. If Mixpanel shows us what users are doing, CleverTap is how we respond — with a push notification, an in-app message, or a survey at exactly the right moment.

The simplest way to understand it

CleverTap is a campaign tool that lets you send the right message to the right user at the right time — automatically. You set the rules once. CleverTap runs the campaign for every user who matches those rules, forever, without you doing anything again.

Why CleverTap specifically?
CleverTap combines user behaviour data, segmentation, and campaign delivery in one tool. It receives real-time events from the MoneyMore app, builds user profiles from those events, and lets you trigger campaigns based on exactly what a user just did — or did not do.

What CleverTap can send

Push notification
A message that appears on the user's phone screen even when the app is not open. This is the highest-reach channel — any user who has granted notification permission can receive it.
e.g. "You've used 80% of your Food & Dining budget this month"
In-app message
A message that appears inside the app while the user is actively using it. Can be a banner, a card, or a full-screen overlay. Only reaches users who are currently in the app.
e.g. A card that appears on the home screen offering to help set a first budget
In-app survey
A one or two question prompt that appears inside the app at a specific moment. Used to collect structured user feedback at high-signal moments — immediately after a user action.
e.g. "Did we get the category right?" — shown after the first transaction is categorised

How CleverTap connects to Mixpanel

Mixpanel and CleverTap are two separate tools that work together. Understanding how they connect is the most important concept in this guide — because a mistake in this connection is what causes campaigns to never fire or to reach the wrong users.

The flow of data

1
A user does something in the MoneyMore app
For example: they successfully save a budget for the first time.
2
The app fires a Mixpanel event
The Budget Created event fires in Mixpanel with its properties — Budget Amount, Category, Status: success.
3
The same event is also sent to CleverTap
Engineering instruments both Mixpanel and CleverTap simultaneously. The same event fires in both tools at the same time.
4
CleverTap checks whether any campaign is triggered by that event
If you have configured a campaign to fire when Budget Created occurs, CleverTap checks whether this specific user matches all the conditions — and sends the campaign if they do.
The most common mistake — and how to avoid it
Configuring a CleverTap campaign based on an event that does not exist in the app, is misspelled, or is firing incorrectly. Before you configure any campaign, open Mixpanel and confirm the trigger event is live and firing correctly. If the event is not in Mixpanel, it is not in CleverTap either.

Event names must match exactly

When you configure a campaign trigger in CleverTap, you type or select the event name. This name must match exactly what engineering instrumented — including capitalisation and spaces. MoneyMore's live events use Title Case with spaces:

Sign Up Completedsign_up_completed · signup completed
Sms Import CompletedSMS Import Completed · sms_import_completed
App Openedapp_opened · AppOpened

Core concepts · Chapter 3

Segments

A segment is a group of users who share something in common. Segments are what make CleverTap powerful — instead of sending the same message to everyone, you send the right message to the right group.

Two types of segments in CleverTap

Live segment
A segment that updates in real time as users meet or stop meeting the criteria. If a user fires the qualifying event, they enter the segment immediately. If they no longer qualify, they leave it.
Live segment: "Users who have fired Budget Created at least once" — updates the moment a new user creates a budget
Static segment (manual)
A snapshot of users who met the criteria at a specific point in time. Does not update automatically. Used for one-off campaigns or exports.
Static segment: "All users who signed up in March 2026" — fixed list, does not change

The segments that matter most for MoneyMore right now

Segment nameCriteriaWhy it matters
Activated users Have fired Sms Import Completed at least once Users who have seen their first value. Base for retention campaigns.
Budget creators Have fired Budget Created with Status = success at least once Highest-retention users. Seed audience for budget-related campaigns.
Non-budget users Have fired Sms Import Completed but have NOT fired Budget Created Largest opportunity for retention lift. Target for budget activation campaigns.
At-risk inactive No App Opened in last 5 days, but had 3+ sessions in prior 7 days Recently active users who are going quiet. Prime target for re-engagement.
New users — day 1 Fired Sign Up Completed in the last 24 hours Onboarding window. First 24 hours are critical for habit formation.
The segment we cannot build yet
We cannot build segments by salary worker, freelancer, or student because the segment user property does not exist in CleverTap or Mixpanel yet. Every segment we build today is based on behaviour, not identity. When user_segment_identified is added in the restructure, we can start personalising campaigns by segment.
Core concepts · Chapter 4

Campaign types

CleverTap has several campaign types. For MoneyMore, we use three: push notifications, in-app messages, and in-app surveys. Each has a different use case, a different reach, and a different level of interruption.

When to use each type

TypeReachUse whenDo not use when
Push notification All users with permission — app does not need to be open User needs to know about something now. Budget alert. Re-engagement nudge. Message is not urgent. User is already in the app.
In-app message Only users currently using the app Surfacing a feature the user has not discovered. Confirming something they just did. User needs to take action when they are not in the app.
In-app survey Only users currently in the app at a specific moment Collecting feedback immediately after a meaningful action. General feedback — surveys must be tied to a specific moment to be useful.

Campaign delivery types

Event-triggered campaign
Fires automatically when a user performs a specific action. Most powerful type — the campaign reaches the user at exactly the right moment.
Trigger: Budget Created → Send: "Budget set. We'll track it automatically and alert you when you're getting close."
Segment-based campaign
Sends to everyone in a defined segment at a scheduled time. Used for broadcasts and rhythmic communications like the weekly spending summary.
Every Monday at 9am → Send to all activated users: "Here's your spending from last week"
Inactivity-triggered campaign
Fires when a user has not performed a specific action within a defined time window. Used for re-engagement and churn prevention.
No App Opened for 5 days → Send: "Quick question — what would bring you back?"
Core concepts · Chapter 5

The global frequency cap

The frequency cap is the single most important rule governing every campaign we send. Understanding it — and never bypassing it — is non-negotiable.

What it is

A frequency cap limits how many messages a user can receive in a given time period. Without a cap, a user could theoretically trigger multiple campaigns in a single day and receive five push notifications — which would cause them to immediately disable notifications or uninstall the app.

MoneyMore global notification rules — never override these
🔴
Maximum 2 push notifications per user per day. Across all campaigns combined. If a user has already received 2 pushes today, no further push campaigns fire for them today regardless of how many they qualify for.
🔴
60-minute gap between a push and an SMS. If a push notification was sent in the last 60 minutes, do not also send an SMS. And vice versa.
🔴
Quiet hours: 11pm–7am. No push notifications or SMS between 11pm and 7am. Schedule campaigns to respect local time zones.
🔴
Maximum 1 survey per user per week across all survey campaigns combined. Set this as a global cap in the CleverTap account settings — not just per campaign.

Where to set the global cap in CleverTap

1
Go to Settings in the left sidebar
Look for the gear icon or Settings menu.
2
Find "Global frequency caps" or "DND settings"
CleverTap calls this DND (Do Not Disturb) in some views. It is the account-level setting that overrides all individual campaign caps.
3
Set push cap to 2 per day and DND hours to 11pm–7am
These settings apply to every campaign in the account. Individual campaign caps can be more restrictive but cannot exceed the global cap.
What happens if you do not set this
Without a global cap, a user who creates a budget, then views their spending summary, then opens the app for the fifth time in one day could receive three separate push notifications in an hour. They will disable notifications permanently. Once a user disables push, you cannot reach them again without them going into their phone settings to re-enable — which almost never happens.
A user fires Budget Created at 2pm, triggering a push. At 3pm they trigger a re-engagement campaign push. At 4pm they trigger a third campaign. What happens?
AAll three pushes are sent — each campaign fires independently
BThe first two pushes send. The third is blocked by the 2-per-day global cap.
COnly the first push sends — one push per day is the rule
DAll three are queued and sent the next morning

Users who sign up but do not connect their bank SMS within 24 hours. This is the most critical activation nudge — without SMS import, the user never sees their spending and the app has no value for them.

The five micro-surveys

Micro-surveys are one-question prompts that appear at specific moments in the app to collect structured feedback. They are not generic "rate us" requests — each one is tied to a precise user action and asks only what is relevant at that moment.

The rule for all five surveys

Survey rules — apply to every survey without exception
📋
Maximum one question per survey — never two questions in the same prompt.
📋
Maximum one survey per user per week — across all five surveys combined. Set globally in CleverTap account settings.
📋
Always dismissible with one tap — never force a response.
📋
Never show during onboarding — wait until the user has completed at least one session.

Our campaigns · Chapter 8

Configuring a survey step by step

This chapter walks you through configuring Survey 3 — the budget confidence check — from start to finish. Survey 3 can be activated right now because Budget Created is already live in Mixpanel.

Before you start — confirm the trigger event is live
Open Mixpanel. Go to the Events report. Search for "Budget Created". Confirm it is showing events in the last 30 days. The volume should be around 170 for a 90-day period. If you see zero events, stop and tell engineering before proceeding.

Configuring Survey 3 in CleverTap

1
Go to Campaigns → Create Campaign → In-App Notification
In-app surveys are configured as in-app notifications in CleverTap. Select "In-App Notification" from the campaign creation menu.
2
Name the campaign: "Survey 3 — Budget Confidence Check"
Use a consistent naming convention so campaigns are easy to find later. Always include the survey number and a plain language description.
3
Set the target segment: users who have fired Budget Created
In the "Who" section, select "Users who have done" → Budget Created → Status = success → at least once. This ensures only users who have successfully created a budget see the survey.
4
Set the trigger: Budget Created with Status = success
In the "When" section, select "When user performs event" → Budget Created → add filter: Status equals "success". The survey fires immediately when a successful budget creation happens.
5
Add the frequency cap: once per user, ever
In the delivery settings, set "User receives this campaign" to "Only once". This is critical — a user who creates a second budget should not see the survey again.
6
Create the message — English version first
Question text: "How confident are you about this limit?" — Answer options as three buttons: "Just guessing" · "Pretty sure" · "Very confident". No follow-up question. Keep the design minimal — one question, three buttons, dismiss option.
7
Create the Arabic version
Question: "ما مدى ثقتك في هذا الحد؟" — Answers: "مجرد تخمين" · "متأكد إلى حد ما" · "واثق جداً". Set text direction to right-to-left. Have a native Arabic speaker review before activating.
8
Set the display timing: show immediately after Budget Created fires
No delay needed — the user is on the confirmation screen. The survey should appear inline below the confirmation message, not as a popup overlay.
9
Test in staging before activating
Use a test device to create a budget in the staging environment and confirm the survey appears correctly, the Arabic version renders right-to-left, and the dismiss button works.
10
Activate and monitor for 48 hours
After activating, check CleverTap analytics after 48 hours to confirm impressions are being recorded. Check Mixpanel to confirm Budget Created is still firing correctly after the campaign goes live.
What to do with the responses
Growth Marketing reviews survey responses every sprint in the signal review session. If more than 40% of responses say "Just guessing" — this is a direct signal that the budget creation screen is not giving users enough context about what a reasonable limit looks like. That finding escalates to the Product backlog as a Fix Brief for the budget creation UX.
Reading & improving · Chapter 9

Reading campaign analytics

Every campaign you send generates data. Reading that data correctly tells you whether the campaign is working, who it is reaching, and what to change.

The four numbers that matter for every campaign

MetricWhat it meansWhat is good
Sent How many users the campaign was sent to Tells you the size of your qualified audience. If lower than expected, check your segment criteria.
Delivered How many of the sent messages actually reached the device Should be above 95% of sent. If it is lower, users may have uninstalled the app or revoked notification permission.
CTR (Click-through rate) Percentage of delivered messages that were tapped For MoneyMore push notifications, above 10% is good. Below 5% means the message or timing needs improvement.
Conversion Percentage of users who completed a defined goal action after receiving the campaign This is the most important metric. Define the goal before the campaign launches — not after.

For survey campaigns — the four additional metrics

MetricWhat it means
ImpressionsHow many times the survey was shown to users
Response ratePercentage of users who answered vs dismissed. Below 30% means the timing or question is wrong.
Answer distributionThe breakdown of how users answered. This is the actual signal you are collecting.
Free text responsesOptional open-text answers. Read these individually — they often contain the most actionable insights.

How to tell if a push notification is underperforming

If a push notification has a CTR below 5% after at least 50 delivered messages, it is underperforming. The most common causes:

1
The message is too generic
Messages with personalised variables (actual amounts, category names, days remaining) consistently outperform generic messages. Add personalisation if it is missing.
2
The timing is wrong
A budget alert sent at 11pm will be ignored. Check what time of day your delivered messages are reaching users and whether it aligns with when they are likely to act.
3
The segment is too broad
A message sent to all users regardless of their behaviour is less relevant than a message sent to users who just did something specific. Tighten the segment before changing the copy.
Reading & improving · Chapter 10

A/B testing

A/B testing lets you send two versions of the same campaign to two groups of users to see which performs better. It is how you improve campaigns based on evidence rather than opinion.

When to run an A/B test

A/B testing only makes sense when you have enough volume to get a statistically meaningful result. For MoneyMore's current user base, this means:

Minimum volume for an A/B test at MoneyMore's current scale
You need at least 200 users per variant to draw a meaningful conclusion. If your campaign is only reaching 100 users total, do not run an A/B test — pick the better option based on judgement and test later when volume is higher.

What to A/B test and what not to

Worth testingNot worth testing yet
Two different message copies for the same campaignCompletely different campaign strategies
Sending at 9am vs 12pm for the same campaignDifferent trigger events
With personalisation vs without personalisationSegment criteria differences
English copy variationsArabic copy variations (get a native speaker to review instead)

How to set up an A/B test in CleverTap

1
Create the campaign as normal — then look for "A/B test" option
CleverTap offers A/B testing within the campaign builder. After setting your audience and trigger, look for the "Create variants" or "A/B test" option before finalising the message.
2
Create variant A and variant B — change only one thing
The golden rule of A/B testing: change exactly one thing between variants. If you change both the copy and the timing, you will not know which change drove the difference.
3
Set the split to 50/50 and define the winning metric before launch
Decide before launching: "The winner is the variant with the higher CTR after 500 delivered messages." Write this down. Do not change the winning metric after you see early results.
4
Wait for the defined volume then declare a winner
Do not check results hourly. Check once you have hit the minimum volume. Then roll out the winning variant to 100% of users and document what you learned.
Log every A/B test result in the Experiment Tracker
Every A/B test result — win, loss, or inconclusive — goes into the Experiment Tracker Excel file. This prevents the team from re-testing things that have already been tested and builds a library of what works for MoneyMore users specifically.
Day-to-day · Chapter 11

Before every campaign

Every campaign — push notification, in-app message, or survey — goes through this checklist before it goes live. No exceptions.

Pre-launch checklist

Trigger event verified in Mixpanel. Confirm the event is live, firing at the correct moment, and carrying the correct properties. Check the last 7 days in the Events report.
Event name matches exactly. Check capitalisation and spaces. Open the CleverTap campaign builder and verify the event name in the trigger matches what you see in Mixpanel character for character.
Segment criteria reviewed. The audience the campaign will reach makes sense for the message. Not too broad, not so narrow that fewer than 50 users qualify.
Frequency cap set. Per-campaign cap is set. Global cap (2 pushes per day, 1 survey per week) is confirmed active in account settings.
Quiet hours respected. Campaign is not scheduled to deliver between 11pm and 7am.
English copy reviewed. Clear, direct, calm. No exclamation marks. No alarming language about money. Personalisation variables use the correct variable names.
Arabic copy reviewed by a native speaker. Not machine translated. RTL layout confirmed. Variable placeholders in correct position.
Deep link tested. If the campaign has a deep link, confirm it opens the correct screen in the app on both Android and iOS (when iOS is live).
Tested on a staging device. The campaign has been triggered on a test device and the message appeared correctly.
Success metric defined. Written down before launch: "This campaign is successful if [metric] reaches [threshold] within [time period]."
Review scheduled. A calendar reminder exists to check campaign analytics 48 hours after launch.
When to escalate before launching
If the trigger event has zero volume in Mixpanel, or if the segment has fewer than 50 users, or if you are not sure whether the event fires at the right moment — stop and bring it to the signal review session before activating. It is better to delay by one sprint than to launch a campaign that wastes user attention on a broken experience.
Day-to-day · Chapter 12

Things to never do

The fastest way to damage the MoneyMore notification channel is to send the wrong message at the wrong time to the wrong person. These are the most important rules.

Never launch a campaign without verifying the trigger event in Mixpanel first
A campaign with a non-existent or broken trigger event will either never fire, or fire at the wrong moment for the wrong users. This wastes user attention — which is a finite resource. Two wasted notifications and a user disables push forever.
Never use placeholder copy in a live campaign
If you see "[category]" or "[amount]" appearing literally in the message rather than the actual value, the personalisation variable is broken. Test every variable in staging before activating. A message that says "You've used 80% of your [category] budget" is worse than no message at all.
Never bypass the global frequency cap
If you think a campaign is "important enough" to send even though a user has already received 2 pushes today — it is not. There are no exceptions to the daily cap. The cap exists to protect the channel. Breaking it costs more long-term than any short-term gain from the extra message.
Never send push notifications during quiet hours
A budget alert at 2am causes two things: the user is annoyed, and they disable notifications. Both outcomes are permanently damaging. Quiet hours are 11pm–7am. Respect them unconditionally.
Never change a campaign while it is live without telling Product
If you edit the copy, segment, or trigger of a live campaign, the data collected before and after the change is no longer comparable. If a campaign needs to change, pause it first, make the change, document what changed and why, then reactivate. Always tell Product before pausing or changing a live campaign.
