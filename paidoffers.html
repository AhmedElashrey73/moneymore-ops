Paid Offers · Paymob checkout
A new claim_type: paid alongside the existing free claim types. User pays via Paymob's hosted checkout (Card, Vodafone Cash, Fawry, InstaPay, BNPL) and on success the code reveals in the standard claim sheet. Single-step model — no extra tap after payment success.
The problem
Until now every offer in the catalog has been free. Partners with high-value deals (70%+ off, bundled value vouchers, time-limited exclusives) want a way to monetize discovery. Paid offers solve this by adding a payment gate before code reveal, using Paymob (Egypt's dominant gateway). The design challenge: make the price visible before tap, never spring a payment screen on a user who expected free, and route success cleanly back to the standard claim flow.

Handoff
Designer reference for Paid Offers v1 (Android). Covers the MoneyMore-authored screens (card, sheet, code reveal, receipt, error toast) and references the Paymob hosted checkout for context — the gateway is integrated by engineering and not designed by us.
What MoneyMore designs · what Paymob designs
  • MoneyMore designs: offer card paid variant (purple gradient hero / purple coupon block / price pill / "Unlock" CTA), detail sheet paid variant (price summary block / "Buy for X EGP" CTA / "What you get" terms), return transition screen, post-payment sheet (code reveal + receipt row), payment failure toast, My Offers paid card.
  • Paymob designs: method selector, card form, 3DS OTP prompt, wallet phone entry, processing screen, success screen, failure screen. The Paymob screens in this prototype are reference only — they show what the user will see, but engineering integrates Paymob's SDK without customising those screens.
  • The handoff between us and Paymob: we redirect out (or open a webview) on "Buy for X EGP" tap; we get a deep-link callback to a known URL with a status code; we reconcile with our backend and render the code reveal or failure toast.

Engineering · critical race conditions to handle
Paid offers introduce a new failure mode: payment succeeded but app closed before code allocation. These are the cases that must be handled or the user pays and gets nothing.
  • Payment succeeded, app killed before return. On next app open, reconcile with Paymob webhook + backend ledger. If a successful payment exists with no claim, allocate code and notify user via push: "Your Elgameya code is ready."
  • Payment ambiguous (timeout, network drop during processing). Treat as "we'll check and let you know" — don't show success or failure. Push the user the result once the backend reconciles.
  • Idempotency on the /claim endpoint. Reopening a purchased offer must NEVER trigger a new /claim or new payment. Bind the code to (user, offer) on first successful payment.
  • Double-tap on Buy for X EGP. Disable the CTA on first tap until the gateway loads — avoid duplicate orders.
  • Refunds are out of scope for v1. If a user reports a problem with a paid offer, customer support handles it manually via Paymob's dashboard. Worth surfacing in the receipt row that this is the case.
  • Save-card UX (Paymob feature). Paymob can store cards for repeat purchases. v1 leaves this disabled in the form (unchecked) — revisit for v2.
Accessibility — what changes what you draw
  • Price pill on the card needs to be announced before the CTA. Screen reader order: discount → merchant → title → price → CTA. So the user knows it's paid before being prompted to tap.
  • "Buy for {price} EGP" CTA must say the full price out loud — not "Buy now" with the price as a separate label. Bundle it.
  • The receipt row is informational, not interactive in v1. If we add "View receipt" detail in v2, that becomes a tappable target needing 44×44px.
  • Failure toast colour vs text. Red toast plus explicit "Payment didn't go through" — never colour alone signalling failure.
  • Paymob screens are not in MoneyMore's a11y scope. Engineering should still test them but Paymob owns those surfaces. If a user reports an a11y issue on Paymob screens, escalate to Paymob.
